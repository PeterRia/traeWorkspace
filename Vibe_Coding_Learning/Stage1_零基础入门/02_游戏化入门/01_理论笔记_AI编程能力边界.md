# 02 游戏化入门 - 理论笔记：AI 编程能力与边界

> 参考：https://datawhalechina.github.io/easy-vibe/zh-cn/stage-1/ai-capabilities-through-games/

## 一、AI 能帮你做到什么程度？

### 可以胜任的范围
- 简单的内部小工具
- 数据可视化看板
- 轻量级小游戏
- 产品原型 / Demo

### 不适合的场景
- 一键生成可直接商用的成熟产品（需要人工打磨）
- 强安全/强合规系统（支付、风控、医疗）
- 需要复杂架构设计的大型系统

### 结论
AI 更适合做"小而清晰"的应用，大型项目需要"流程视角"——你负责拆步骤和架构，AI 负责加速每个小步骤。

## 二、AI 编程的典型模式

### 成功模式
1. 范围清晰：一页网页、一个简单工具、一个小游戏
2. 结果可见：能立即在浏览器中验证
3. 纠错直接：发现问题后用对话修正

### 你需要做的
- 把整体需求拆成清晰的步骤
- 理解每一步的输入输出
- 负责架构设计、系统集成和运维
- 在原型/内部工具级别，AI 可以打第一版

## 三、对话编程能做什么不能做什么

| 能做 | 不能做（或需要人工参与） |
|------|----------------------|
| 生成完整网页 | 一键生成商用级产品 |
| 实现基本交互逻辑 | 复杂业务规则处理 |
| 快速迭代修复 bug | 性能优化和架构设计 |
| 理解并实现具体需求 | 理解模糊的、未定义的需求 |

## 四、AI 编程平台对比

| 平台 | 特点 | 适用场景 |
|------|------|---------|
| z.ai | 全栈开发，实时预览 | 零基础入门 |
| Google AI Studio | 谷歌出品，Gemini 模型 | 快速原型 |
| Figma Make | 与设计工具整合 | 设计师快速实现 |
| Coze | 零代码可视化搭建 | 对话应用/助手 |

| Lovable | 高质量 React 应用 | 前端应用开发 |
| Replit Agent | 在线 IDE + AI 助手 | 多语言协作开发 |

## 五、提示词技巧

### 好的提示词特征
- **具体明确**：不只说"做个游戏"，说明具体功能和交互
- **分步骤**：不一次要全部功能，逐步迭代
- **有约束条件**：指定技术栈、设计风格、目标用户

### 示例对比

**差的提示词**：
```
做个贪吃蛇游戏
```

**好的提示词**：
```
帮我做一个贪吃蛇游戏：
1. 用方向键控制蛇的移动
2. 吃到食物后蛇会变长，分数增加
3. 撞到墙壁或自己的身体就游戏结束
4. 要有开始和重新开始按钮
5. 界面要简洁好看
```

## 六、遇到问题的处理策略

1. **复制错误信息**：把完整的报错堆栈粘贴给 AI
2. **截图描述**：UI 问题直接截图 + 描述期望效果
3. **多轮对话**：不要一次就放弃，持续追问
4. **换个角度描述**：同样的需求，换种说法可能效果更好

| 术语 | 含义 |
|------|------|
| z.ai | 智谱 AI 的全栈开发平台 |
| 全栈开发 | 同时涉及前端（界面）和后端（服务器）的开发 |
| 原型 | 可以展示核心功能但未完善的初版产品 |
| 迭代 | 逐步修改、逐步完善的过程 |
